## 3. R-5_2nsF_ZM — 個人エージェント vs 本番エージェントの基盤（t=940 周辺含む）

### メタ情報
- **タイトル**: I Love the Karpathy LLM Wiki but it Doesn't Scale. Here's What Does.
- **チャンネル**: Cole Medin
- **長さ**: 18:57 / **公開日**: 2026-07-09
- **字幕**: 英語 auto-subs

### 要旨
- 全体テーゼ: **markdown 駆動の個人エージェント（Karpathy LLM Wiki 型第二の脳）は自分専用なら最適だが、他人に提供する production エージェントには一切スケールしない**。多ユーザー・ライブデータ・アクセス制御・監査可能性・コスト最適化が要る瞬間に DB ベースの「コンテキスト層」アーキテクチャへ転換せよ。
- 本番アーキテクチャは 2 部品: ① **Context Retriever** — 無構造 DB（デモは Redis の key-value）の上にエンティティ定義（customer/order/product…）＝スキーマ層を宣言すると、属性フィルタ・全文検索などの **MCP ツールが自動生成**され、エージェントは index 文書を読む代わりに単一ツールコールでデータ取得（1000 トークン未満）。② **Agent Memory** — 会話ごとの短期記憶を保存し、バックグラウンドプロセスが要点を長期記憶へ自動昇格、vector/semantic search で per-user 数百万件の記憶から関連分だけ recall。
- **t=940（15:40）付近の内容**: まさに②の核心部。短期→長期への「golden nugget 抽出・昇格」は個人エージェントの memory.md と同じ発想を DB+vector で本番スケール化したもの、という説明。ユーザーが明示的に「覚えて」と言わなくても抽出される、記憶の削除も可能、記憶管理層はプラットフォーム側が担うがデータは全て自分の DB 内で可視、と続く。
- コーディングエージェント SDK（Claude Agent SDK 等）は長時間タスク向けで遅くトークン重量級なので本番エージェントには Pydantic AI を推奨、という実装ノートも。Redis（スポンサー協業）の Redis Iris を実演台に使用。

### ver3 への関連・採否所見
- ユーザーの推測（「人間がブラウズする代わりに各人の代理 AI がデータを取りに来る／置きに来る」）と動画の主張は**方向一致**。動画の言い方では「エージェントが読む前提なら、markdown を読ませるのではなく、エンティティ＝スキーマを宣言してツール（MCP）経由で構造化アクセスさせろ」。
- ver3 への写像: ver3 は既に schemas/ 正本 + AI ファーストデータ設計を持つ。この動画から採れる具体アイデアは「**schemas/ のエンティティ定義から研究データ用の read 系 API/MCP ツールを機械生成し、外部ユーザーの代理エージェントが index 文書を読まずに 1 コールで取得できる面を用意する**」という将来形。Truth append-only とも矛盾しない（read は自由、write は INSERT のみのツールにすればよい）。
- ただし Redis 常駐 DB・vector メモリ基盤は**不変条項①（常駐 DB を SSOT にしない・10 年コスト最小）に正面から抵触**するため製品としては不採用。**採否: 思想（エージェント向け構造化アクセス面・MCP ツール自動生成）は採る価値あり、実装スタック（Redis/常駐 memory 基盤）は不採用**。Phase 後半の「AI 入口 API」設計時に再参照推奨。

### 要旨
- 内容は UI/UX の原理教材というより「**AI 生成デザインの均質化（AI slop）を、用途別スキルの組合せで破る**」カタログ動画。核となる原則は以下。
  1. **デザイン方向の事前コミット**: モデルは学習分布の最頻手（紫グラデ・定番フォント）に流れる。Anthropic frontend-design スキルはコードを書く前に方向性を 1 つ選ばせ一貫させることで generic 化を防ぐ。ただしこれはマーケ/LP 向けで、**機能 UI には不向き**。
  2. **機能 UI は既製コンポーネント**: ダッシュボード等は shadcn の registry から professional-grade 部品を引く（スキル=ルールブック＋MCP=部品取得の 2 段構え。スキルは必要時のみロードされ常駐 MCP よりコンテキスト効率が良い）。
  3. **ダッシュボードは配置の問題**: 部品の一貫性より「情報のグルーピングと 1 画面の情報量」を先に推論させる専用 dashboard スキル。
  4. **業種適合の決定エンジン**: UI UX Pro Max は 161 業種 DB を検索し、色・フォント・レイアウトを業種に合わせて選定してからテーラーメイドのデザインシステムをモデルに渡す（「良いデザイン一般」ではなく用途特化の決定を先に与える）。
  5. **アニメーションは GSAP スキル**: AI は scroll-reveal 一発芸に陥りがち＋size/position 変更でブラウザ再レイアウトを起こし janky になる。GSAP 公式スキルは compositor に優しい動きへ誘導。
  6. その他: テイストプリセット（minimalist/brutalist/premium 等は 1 つだけ選ぶ）、モバイルは「小さい Web ではない」（thumb zone・Material 3・SwiftUI/liquid glass・Expo）。
- スポンサー枠 2 件（SciSpace、Higgs Field）あり。

### ver3 への関連・採否所見
- 「これで UI を学べば最良の UX が作れるか」への答えは**部分的に No**。この動画は UX 原理の教材ではなく Claude 用スキルの選定ガイド。普遍原理として持ち帰れるのは「方向性を先に決めさせる」「機能 UI は既製部品」「ダッシュボードは情報配置から」「モバイルは別物」の 4 点で、いずれも本環境導入済みの `impeccable` スキル＋`dataviz` スキルの守備範囲とほぼ重複。
- ver3 は UI-as-data（ScreenDef JSON→React レンダラ）なので、画面ごとに AI が自由デザインする前提の本動画とは構造が逆。活きるとすれば「**レンダラ側のデザイントークン/コンポーネント標準を shadcn 系の既製品に寄せ、LP 等のマーケ面だけ方向性コミット型スキルを使う**」という切り分け方。**採否: 動画自体の追加導入は不要。上記 4 原則のみ設計メモとして採録**。
