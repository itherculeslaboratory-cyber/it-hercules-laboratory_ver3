# ADR: 旧設計(legacy)経済カーネルと現行モデルの整合(V3-KRM-18)

## 状況(Context)

要件 V3-KRM-18(`01-requirements/registry.json` `superseded` 注記付き):
「(旧設計/legacy経済カーネル) カルマ計算(通常違反+1/重大違反+5)、貢献度100ごとに
プラチナコイン付与・10000で称号、フィボナッチ価格の免罪符、30日無違反でkarma+10・
eventCount-1の月次自動軽減を行う経済カーネルとしてルールをGUI編集可能にする。」

`superseded` 注記: 「通常+1/重大+5・30日軽減の旧数値ルールは Fibonacci二層カルマ
(V3-KRM-01/02/03)へ改定済み。整合(RECONCILE)は要確認」。本 ADR がその確認を行い、
現行実装のどの部分が旧設計を正式に置き換え済みか(=「現行が正」)を対応表として固定する。

## 対応表(旧設計 → 現行実装・「現行が正」)

| # | 旧設計(legacy) | 現行実装(ver3) | 判定 |
|---|---|---|---|
| 1 | カルマ計算: 通常違反 +1 / 重大違反 +5(固定加点・カルマ値を直接動かす) | **Fibonacci 二層モデル**(V3-KRM-01/02): カルマは「カルマ値[-100,+100]」と「カルマカウント」の二層。違反はカウントを+n進め、カウント遷移ごとに `-Fib(n)` を value へ減点(`apps/api/src/ledger-routes.ts` の `fib`/`fibPenalty`/`grantKarmaCountIncrease`)。固定 +1/+5 という単純加点は廃止。 | **現行が正**。初犯を軽く・累犯を急激に重くする設計意図(検算アンカー: 0→5 で -12, 5→10 で -131・`economy-constants.ts` `FIB_PENALTY_ANCHOR_*`)を旧固定加点モデルは表現できない。 |
| 2 | 貢献度 100 ごとにプラチナコイン付与・10000 で称号(単一軸・固定 100 閾値) | **3 軸独立モデル**(V3-KRM-10/11/12): research/capital/development を非統合で独立追跡。軸ごとに累計 100 pt→1PT、2 枚目以降は `Fib(n)*100` で閾値が増加(`apps/api/src/contribution.ts` `mintFromScore`)。称号閾値 10000 は軸ごとに投影導出(`CONTRIBUTION_TITLE_THRESHOLD`)。 | **現行が正(称号閾値 10000 は一致・単一軸/固定閾値の部分のみ置換)**。「100 ごとに1枚」は 1 枚目のみ旧設計と一致し、2 枚目以降の Fibonacci 逓増は round-16 以前の ADR-H-38 確定(`V3-KRM-12`)で追加された正式な改定。 |
| 3 | フィボナッチ価格の免罪符 | **据置(そのまま現行)**: V3-KRM-05 `indulgenceStage`(`apps/api/src/contribution.ts`)が初回 `fib(1)=1PT`・購入毎+1段階・暦月毎-1段階(下限1)を実装。旧設計との数値上の矛盾なし。 | **現行が正 = 旧設計と同一設計を正式踏襲**(改定ではなく継承)。 |
| 4 | 30日無違反で karma+10・eventCount-1 の月次自動軽減(1回の軽減で value+10 と count-1 が同時に起きる・30日ローリング窓) | **KRM-03 月次カルマ救済**(`apps/api/src/batch.ts` `krmMonthlyRecovery`): 毎月25日(クレジット締め日型)基準・**当月内で相互排他**: (a) 当月中に違反(count 増加)が一度もなければ value+10(上限100・count は不変) / (b) 当月中に違反があれば count-1 のみ(value 救済なし)。30日ローリングではなく UTC 暦月+固定25日基準。 | **現行が正**。旧設計は「value+10 と count-1 が常に同時に起きる」という設計だが、これは「軽減」と「回復」を混同しており、現行の相互排他モデル(V3-KRM-03 確定要件文「期間中1日でもcount≥1ならその月は+10なし」)の方が要件定義書 v1.10 の確定文言と直接一致する。数値(30日→25日基準の暦月)・トリガ条件(同時発生→排他)の両方で現行が優先。 |
| 5 | ルールをGUI編集可能にする(経済カーネル全体) | **V3-KRM-16 policy table**: `config/economy-policy.csv` + `resolvePolicyInt`(`apps/api/src/policy.ts`)が `policy_key/policy_int/timestamp` の append-only 履歴で GUI 編集相当の可変パラメータ管理を実装(称号閾値・免罪符初期価格・カルマ上限/BAN閾値が対象)。実際の GUI 画面(フォーム)自体は後波(design-k3 §2.7 注記どおり)。 | **現行が正(データ層は完了・画面 UI は既知の残課題として別途)**。 |

## 決定(Decision)

1. 上記 5 項目はいずれも現行実装(Fibonacci 二層カルマ・3 軸貢献度・免罪符・月次救済・
   policy table)が正式かつ最新の設計である。旧設計の数値(+1/+5・30日・同時発生)を
   参照実装として復元する作業は行わない。
2. 免罪符(項目3)は例外的に「改定」ではなく「継承」であり、旧設計の記述自体が現行と
   矛盾しない(混同を避けるため表に明記)。
3. GUI 編集可能性(項目5)は「データ層(policy table)」と「画面(フォーム UI)」を分離し、
   前者を V3-KRM-18 の充足範囲、後者を別要件(V3-GOV-17 管理 GUI・後波)の担当とする。
4. 本 ADR 自体が V3-KRM-18 の完了物(整理文書+実装済み対応表)であり、追加コードは
   発生しない(旧設計はどこにも実装されておらず削除対象のコードも無い — 純粋な文書
   整理タスク)。

## 結果(Consequence)

- V3-KRM-18 の `superseded` 注記が要求する「整合(RECONCILE)の確認」は本 ADR で完了。
- 今後 V3-KRM-01/02/03/05/11/12/16 のいずれかを変更する際は、本 ADR の対応表を
  「なぜ現行の数値/条件が正なのか」の参照点として使う。
